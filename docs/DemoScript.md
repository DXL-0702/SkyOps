# SkyOps Agent 初赛 Demo 演示脚本

本文档用于初赛现场演示排练。目标不是把每个功能都讲完，而是在有限时间内让评委清楚看到：

- SkyOps Agent 不是无人机缺陷识别工具，而是低空作业任务自治与风险推演系统。
- Demo 展示的是“任务输入 -> 方案生成 -> 风险解释 -> 异常重规划 -> 复盘评测”的闭环。
- 当前版本使用 mock/simulated data，不控制真实无人机，不替代审批和人工安全责任人。
- 系统能力可以被测试、被量化、被复盘，而不是只停留在概念描述。

## 1. 演示前准备

### 1.1 推荐页面顺序

演示时建议按以下顺序切换界面：

1. Mission Operations Console 初始任务输入页。
2. 点击生成任务方案。
3. Mission Plan 任务方案页。
4. Risk 风险解释页。
5. Evidence / Decision Trace 决策依据页。
6. Incident 异常注入与重规划页。
7. Mission Review 任务复盘页。
8. Evaluation Summary 评测摘要页。

### 1.2 推荐输入任务

```text
明天上午巡检南山区一栋180米高办公楼外立面，重点排查幕墙裂缝和脱落风险，要求尽量减少对行人的影响。
```

### 1.3 推荐注入异常

优先选择：

```text
wind_speed_spike
```

中文讲解口径：

```text
我们现在模拟任务执行中风速突然升高，看 Agent 是否能主动重规划，而不是继续执行原方案。
```

### 1.4 关键边界声明

演示开始或进入技术细节前必须说明：

```text
当前 Demo 使用的是 mock/simulated data，不接真实无人机、不控制真实飞行、不替代飞行审批和人工安全责任人。我们重点展示的是任务级自治决策层的工程闭环。
```

## 2. 6 分钟完整版脚本

### 0:00 - 0:30 开场定位

操作：

- 打开 Mission Operations Console。
- 停留在任务输入页，不要先展示已经生成的方案。

台词：

```text
各位老师好，我们展示的是 SkyOps Agent，一个面向深圳低空经济场景的低空作业任务自治与风险推演智能体。

它的产品本体不是无人机缺陷识别，也不是底层飞控系统。我们要解决的问题不是“图像里有没有裂缝”，也不是“无人机会不会飞”，而是一次低空作业任务在执行前、执行中、执行后，应该如何安全、可解释、可复盘地完成。

简单说，SkyOps Agent 关注的是：这个任务该不该飞，什么时间飞最安全，如何规划起降点和航线，遇到异常时如何重规划，以及任务结束后如何形成复盘闭环。
```

### 0:30 - 1:10 自然语言任务输入

操作：

- 在输入框中展示或输入推荐任务。
- 指向任务输入区域和场景标注。

台词：

```text
在用户体验上，我们不希望用户像配置工业软件一样填写大量表单，而是像对专业低空调度员说话。

这里我输入一个深圳高层建筑外立面巡检任务：明天上午巡检南山区一栋180米高办公楼外立面，重点排查幕墙裂缝和脱落风险，要求尽量减少对行人的影响。

注意，这里的“巡检”只是第一个 Demo 场景。系统真正抽象的是低空作业任务，可以扩展到园区光伏、工地安全、应急救援、园区安防、消防巡查、低空测绘和物流配送等场景。
```

操作：

- 点击“生成任务方案”。

台词：

```text
点击生成后，系统会把自然语言任务转成结构化任务上下文，并结合模拟的天气、人流、空域、设备状态和安全规则进行推理。
```

### 1:10 - 2:00 任务方案生成

操作：

- 切换到 Mission Plan 或任务方案区域。
- 展示推荐时间窗口、起降点、航线策略、安全阈值、中止条件。

台词：

```text
这里生成的是任务级方案，而不是一份普通的巡检报告。

系统会给出推荐执行时间、推荐起降点、航线策略、安全距离、拍摄间距、任务拆分方案，以及关键的安全阈值和中止条件。

例如，它不会只说“可以飞”，而是会解释为什么选择这个时间窗口，为什么需要避开人流高峰，为什么要设置返航电量和风速阈值。

这体现了 SkyOps Agent 的核心能力：主动感知、多源约束推理、自主规划和可解释决策。
```

可补充说明：

```text
当前阶段的天气、人流、空域和设备数据是 mock/simulated data，界面和文档中都会明确标注。我们不会把模拟数据伪装成真实数据。
```

### 2:00 - 2:50 风险解释

操作：

- 切换到 Risk 风险解释页。
- 展示风险等级、风险项、触发条件、建议动作。

台词：

```text
接下来是风险解释部分。传统系统往往只给用户一个结果，但低空作业更需要知道为什么。

SkyOps Agent 会把风险拆成可解释的条目，例如风速、GPS遮挡、人流聚集、图传延迟、电量余量、临时限飞区等。

每个风险项都包含来源、触发条件、影响范围和建议动作。这样后续无论是前端展示、人工复核，还是评测系统打分，都有明确依据。

我们的安全原则是保守的：当信息不足或风险不确定时，系统应该建议人工复核、暂停执行或启用保守方案，而不是冒险推进。
```

### 2:50 - 3:30 决策依据与可解释性

操作：

- 切换到 Evidence / Decision Trace 或相关解释区域。
- 展示事实输入、模型推理、建议动作、人工确认项的区分。

台词：

```text
这里我们特别强调“事实输入”和“模型推理”的区分。

比如天气、人流、设备电量属于输入条件；风险等级和任务拆分建议属于系统推理；返航、绕行、补飞、人工接管则属于建议动作。

这对于低空作业非常重要，因为系统不能替代法规审批，也不能替代最终安全责任人。它应该把判断依据讲清楚，让人可以审查、接管和复盘。
```

### 3:30 - 4:35 异常注入与重规划

操作：

- 切换到 Incident 异常处置页。
- 注入 `wind_speed_spike`。
- 展示重规划动作，例如暂停任务、返航、切换备用航线、拆分补飞、保留已采集数据。

台词：

```text
下面我们模拟执行中的突发异常：风速突然升高。

低空作业的关键不是计划永远不变，而是计划遇到异常时能否安全调整。

在这个事件发生后，Agent 会重新评估当前风险等级，并给出重规划动作。例如暂停当前航段、触发返航或切换备用方案，同时保存已经采集的数据，生成后续补飞建议。

这里体现的是异常重规划能力。系统不是固定流程表单，也不是只会聊天的机器人，而是在约束变化后主动更新任务决策。
```

如果界面显示具体重规划原因，可以补充：

```text
可以看到，系统不是简单报错，而是说明触发条件、风险变化、处置动作和剩余任务如何处理。
```

### 4:35 - 5:15 任务复盘闭环

操作：

- 切换到 Mission Review 任务复盘页。
- 展示任务完成度、数据质量评分、风险触发记录、未覆盖区域、补飞建议、下次优化建议。

台词：

```text
任务结束后，SkyOps Agent 会生成复盘结果。

复盘不是简单生成报告，而是记录任务完成度、数据质量、风险触发过程、异常处置过程、未覆盖区域和补飞建议。

这让每一次低空作业不再是一次性执行，而是形成可复盘、可优化的运营闭环。长期来看，系统可以沉淀不同场景下的任务经验和安全策略。
```

### 5:15 - 5:45 评测摘要

操作：

- 切换到 Evaluation Summary。
- 展示评测用例数量、硬约束通过率、风险召回率、异常处置得分、可解释性得分。

台词：

```text
最后是评测部分。我们认为 Agent 项目不能只讲概念，必须可测试、可量化。

所以我们构建了低空任务仿真评测集，用多个 mock/simulated 场景测试 Agent 在复杂约束下的决策能力。

评测指标包括硬约束通过率、风险识别召回率、方案效率、异常处置得分和可解释性得分。

也就是说，我们测试的不是无人机硬件，也不是 CV 缺陷识别，而是 Agent 是否能在低空作业约束下做出安全、合理、可解释的任务决策。
```

### 5:45 - 6:00 收尾

操作：

- 回到总览页或停留在评测摘要页。

台词：

```text
总结一下，SkyOps Agent 的核心价值是把低空作业从“能飞”推进到“会运营”。

当前 Demo 已经打通了自然语言任务输入、任务规划、风险推演、异常重规划、任务复盘和评测闭环。

后续我们会在不改变安全边界的前提下，扩展真实天气、地图、空域平台、无人机系统和 LLM adapter，但硬安全规则仍然由显式规则和人工复核兜底。

谢谢各位老师。
```

## 6. 常见问题回答

### Q1: 为什么没有接真实 LLM API？

推荐回答：

```text
我们当前阶段先完成了 LLM adapter 的接口边界和 mock provider，而不是直接绑定某一个真实模型。

原因是低空作业涉及安全和合规，LLM 不能作为硬安全决策的唯一依据。我们的设计是：硬约束和安全阈值由显式规则兜底，LLM 主要用于自然语言理解、解释生成和报告润色。

这样既能体现 Agent 能力，也能保证后续替换不同模型时不影响核心安全逻辑。
```

### Q2: 为什么使用 mock/simulated data？

推荐回答：

```text
初赛阶段我们重点验证任务级自治决策闭环，而不是验证真实无人机接入。

真实天气、空域、人流、无人机和机库系统都可以通过 adapter 扩展接入，但在比赛 Demo 中直接依赖外部真实系统会增加不可控因素。

因此我们先用 mock/simulated data 构建可复现、可测试的评测环境，并在界面和文档中明确标注数据来源，不把模拟数据伪装成真实数据。
```

### Q3: 你们和传统无人机巡检系统有什么区别？

推荐回答：

```text
传统无人机巡检系统通常关注采集图像和识别缺陷，例如裂缝、热斑、破损等。

SkyOps Agent 关注的是更上层的任务自治决策：任务该不该飞、什么时候飞、如何规划、异常时如何处置、任务后如何复盘。

缺陷识别可以作为下游分析能力接入，但它不是 SkyOps Agent 的产品本体。
```

### Q4: 如果系统判断错误怎么办？

推荐回答：

```text
我们的设计不让系统替代人工安全责任人。所有关键决策都要保留解释字段，并区分事实输入、模型推理、建议动作和需要人工确认的事项。

当数据不足或风险不确定时，系统会建议人工复核、暂停执行或启用保守方案，而不是冒险执行。
```

### Q5: 这个项目如何证明 Agent 能力？

推荐回答：

```text
我们不是只做一个聊天界面，而是把 Agent 能力落到可执行闭环里。

系统能够从自然语言任务中抽取约束，综合天气、人流、空域、设备和安全规则生成方案，进行风险推演，遇到异常后重规划，并在任务后复盘。

同时我们建立了仿真评测集，用硬约束通过率、风险召回率、异常处置得分和可解释性得分量化 Agent 的表现。
```

## 7. 现场演示注意事项

1. 不要把项目讲成“无人机巡检缺陷识别系统”。
2. 不要承诺真实飞行、真实审批、真实空域数据或真实无人机控制。
3. 不要说系统可以绕过禁飞区、降低安全余量或替代安全责任人。
4. 如果页面数据来自 mock/simulated scenario，必须明确说明。
5. 如果被问到后续落地，强调 adapter 预留，而不是当前已经真实接入。
6. 如果时间紧张，优先展示“输入任务 -> 生成方案 -> 风险解释 -> 异常重规划 -> 评测摘要”。
7. 现场讲解要少讲技术名词堆叠，多讲任务闭环和安全价值。
