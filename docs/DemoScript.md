# SkyOps Agent 初赛 Demo 演示脚本

本文档用于初赛视频录制和现场排练。由于视频展示需要控制在 5 分钟以内，主脚本按 **4 分 40 秒** 设计，预留约 20 秒给页面加载、切换和口误调整。

演示目标不是把所有功能逐项讲完，而是在短时间内让评委明确看到：

- SkyOps Agent 不是无人机缺陷识别工具，而是低空作业任务自治与风险推演系统。
- Demo 展示的是“任务输入 -> 方案生成 -> 风险解释 -> 异常重规划 -> 任务复盘 -> 评测量化”的闭环。
- 当前版本使用 mock/simulated data，不控制真实无人机，不替代审批和人工安全责任人。
- 系统能力可以被测试、被量化、被复盘，而不是只停留在概念描述。

## 1. 录制前准备

### 1.1 推荐录制顺序

5 分钟视频里不要展示过多页面。推荐只按以下主线录制：

1. Mission Operations Console 初始任务输入页。
2. 点击生成任务方案。
3. Mission Plan 任务方案页。
4. Risk 风险解释页。
5. Incident 异常注入与重规划页。
6. Mission Review 任务复盘页。
7. Evaluation Summary 评测摘要页。
8. 最后回到 Console 总览或停留在 Evaluation Summary 收尾。

Evidence / Decision Trace 如果界面切换顺畅，可以在 Risk 页面中用一句话带过；不要单独展开成一个长段落。

### 1.2 推荐输入任务

```text
明天上午巡检南山区一栋180米高办公楼外立面，重点排查幕墙裂缝和脱落风险，要求尽量减少对行人的影响。
```

### 1.3 推荐注入异常

优先选择：

```text
wind_speed_spike
```

中文讲解口径：

```text
我们现在模拟任务执行中风速突然升高，看 Agent 是否能主动重规划，而不是继续执行原方案。
```

### 1.4 录制边界声明

演示开始 30 秒内必须说明：

```text
当前 Demo 使用 mock/simulated data，不接真实无人机、不控制真实飞行、不替代飞行审批和人工安全责任人。我们重点展示的是任务级自治决策层的工程闭环。
```

## 2. 4 分 40 秒正式录屏脚本

### 0:00 - 0:25 开场定位

操作：

- 打开 Mission Operations Console。
- 停留在任务输入页，不要先展示已经生成的方案。

台词：

```text
各位老师好，我们展示的是 SkyOps Agent，一个面向深圳低空经济场景的低空作业任务自治与风险推演智能体。

它不是无人机缺陷识别工具，也不是底层飞控系统，而是任务级自治决策层。我们关注的问题是：一次低空作业该不该飞、什么时候飞、怎么规划、遇到异常怎么重规划，以及任务结束后如何复盘。
```

### 0:25 - 0:55 说明 Demo 边界与任务输入

操作：

- 指向任务输入框。
- 输入或展示推荐任务。

台词：

```text
当前 Demo 使用 mock/simulated data，不控制真实无人机，也不替代审批和人工安全责任人。

现在我们输入一个深圳南山区高层建筑外立面巡检任务：明天上午巡检一栋180米高办公楼外立面，重点排查幕墙裂缝和脱落风险，并尽量减少对行人的影响。
```

操作：

- 点击“生成任务方案”。

台词：

```text
点击生成后，系统会把自然语言任务转成结构化任务上下文，并结合模拟的天气、人流、空域、设备状态和安全规则进行推理。
```

### 0:55 - 1:40 展示任务方案

操作：

- 切换到 Mission Plan 任务方案页。
- 展示推荐时间窗口、起降点、航线策略、安全阈值、中止条件。

台词：

```text
这里生成的是任务级方案，而不是传统巡检报告。

系统会给出推荐执行时间、起降点、航线策略、安全距离、拍摄间距、任务拆分方案，以及风速、电量、图传和返航相关的安全阈值。

重点是，系统不只说“可以飞”或“不可以飞”，还会解释为什么这样规划，例如为什么避开人流高峰、为什么设置返航电量、为什么需要保守航线。
```

### 1:40 - 2:20 展示风险解释

操作：

- 切换到 Risk 风险解释页。
- 展示风险等级、风险项、触发条件、建议动作。

台词：

```text
接下来是风险推演。低空作业不是单一因素决策，系统需要同时考虑风速、GPS遮挡、人流聚集、图传延迟、电量余量、空域合规等多源约束。

每个风险项都保留触发条件、影响解释和建议动作，便于人工复核，也便于后续评测。

我们的安全策略是保守的：当信息不足或风险不确定时，系统应该建议人工复核、暂停执行或启用保守方案，而不是冒险推进。
```

### 2:20 - 3:10 注入异常并展示重规划

操作：

- 切换到 Incident 异常处置页。
- 注入 `wind_speed_spike`。
- 展示重规划动作，例如暂停任务、返航、切换备用航线、拆分补飞、保留已采集数据。

台词：

```text
现在我们模拟执行中的突发异常：风速突然升高。

低空作业的关键不是计划永远不变，而是计划遇到异常时能否安全调整。

事件发生后，Agent 会重新评估风险等级，并给出重规划动作，例如暂停当前航段、触发返航、切换备用方案，或者把剩余区域拆分为补飞任务。

这体现了 SkyOps Agent 的异常重规划能力：它不是固定流程表单，而是在约束变化后主动更新任务决策。
```

### 3:10 - 3:45 展示任务复盘

操作：

- 切换到 Mission Review 任务复盘页。
- 展示任务完成度、数据质量评分、风险触发记录、未覆盖区域、补飞建议。

台词：

```text
任务结束后，系统会生成复盘结果。

复盘不只是生成一份文字报告，而是记录任务完成度、数据质量、风险触发过程、异常处置过程、未覆盖区域和补飞建议。

这样每一次低空作业都能形成可解释、可复盘、可持续优化的闭环。
```

### 3:45 - 4:20 展示评测摘要

操作：

- 切换到 Evaluation Summary。
- 展示评测用例数量、硬约束通过率、风险召回率、异常处置得分、可解释性得分。

台词：

```text
最后是评测部分。我们认为 Agent 项目不能只讲概念，必须可测试、可量化。

因此我们构建了低空任务仿真评测集，用多个 mock/simulated 场景测试 Agent 在复杂约束下的决策能力。

评测指标包括硬约束通过率、风险识别召回率、异常处置得分和可解释性得分。我们测试的不是无人机硬件，也不是 CV 缺陷识别，而是 Agent 是否能做出安全、合理、可解释的任务决策。
```

### 4:20 - 4:40 收尾总结

操作：

- 停留在 Evaluation Summary 或回到总览页。

台词：

```text
总结来说，SkyOps Agent 的核心价值是把低空作业从“能飞”推进到“会运营”。

当前 Demo 已经打通自然语言任务输入、任务规划、风险推演、异常重规划、任务复盘和评测闭环。

后续系统可以扩展真实天气、地图、空域平台、无人机系统和 LLM adapter，但硬安全规则仍然由显式规则和人工复核兜底。谢谢各位老师。
```

## 6. 常见问题回答

本节主要用于答辩或视频文案准备，不建议完整放进 5 分钟视频。

### Q1: 为什么没有接真实 LLM API？

推荐回答：

```text
我们当前阶段先完成了 LLM adapter 的接口边界和 mock provider，而不是直接绑定某一个真实模型。

原因是低空作业涉及安全和合规，LLM 不能作为硬安全决策的唯一依据。我们的设计是：硬约束和安全阈值由显式规则兜底，LLM 主要用于自然语言理解、解释生成和报告润色。

这样既能体现 Agent 能力，也能保证后续替换不同模型时不影响核心安全逻辑。
```

### Q2: 为什么使用 mock/simulated data？

推荐回答：

```text
初赛阶段我们重点验证任务级自治决策闭环，而不是验证真实无人机接入。

真实天气、空域、人流、无人机和机库系统都可以通过 adapter 扩展接入，但在比赛 Demo 中直接依赖外部真实系统会增加不可控因素。

因此我们先用 mock/simulated data 构建可复现、可测试的评测环境，并在界面和文档中明确标注数据来源，不把模拟数据伪装成真实数据。
```

### Q3: 你们和传统无人机巡检系统有什么区别？

推荐回答：

```text
传统无人机巡检系统通常关注采集图像和识别缺陷，例如裂缝、热斑、破损等。

SkyOps Agent 关注的是更上层的任务自治决策：任务该不该飞、什么时候飞、如何规划、异常时如何处置、任务后如何复盘。

缺陷识别可以作为下游分析能力接入，但它不是 SkyOps Agent 的产品本体。
```

### Q4: 如果系统判断错误怎么办？

推荐回答：

```text
我们的设计不让系统替代人工安全责任人。所有关键决策都要保留解释字段，并区分事实输入、模型推理、建议动作和需要人工确认的事项。

当数据不足或风险不确定时，系统会建议人工复核、暂停执行或启用保守方案，而不是冒险执行。
```

### Q5: 这个项目如何证明 Agent 能力？

推荐回答：

```text
我们不是只做一个聊天界面，而是把 Agent 能力落到可执行闭环里。

系统能够从自然语言任务中抽取约束，综合天气、人流、空域、设备和安全规则生成方案，进行风险推演，遇到异常后重规划，并在任务后复盘。

同时我们建立了仿真评测集，用硬约束通过率、风险召回率、异常处置得分和可解释性得分量化 Agent 的表现。
```

## 7. 视频录制注意事项

1. 主视频控制在 4 分 40 秒左右，不要卡满 5 分钟。
2. 不要把项目讲成“无人机巡检缺陷识别系统”。
3. 不要承诺真实飞行、真实审批、真实空域数据或真实无人机控制。
4. 不要说系统可以绕过禁飞区、降低安全余量或替代安全责任人。
5. 如果页面数据来自 mock/simulated scenario，必须明确说明。
6. 如果录制时页面加载较慢，优先删减“风险解释”和“复盘”中的解释句，保留异常重规划和评测摘要。
7. 如果必须进一步压缩，优先展示“输入任务 -> 生成方案 -> 风险解释 -> 异常重规划 -> 评测摘要”。
8. 录制时少讲技术名词堆叠，多讲任务闭环和安全价值。
