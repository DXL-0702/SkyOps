## 核心协作原则
1.  **禁止自主决策**：在项目开发全流程中，**绝不自作主张猜测我的需求、意图或技术选型**。任何需要推进的开发步骤、功能调整、代码修改、技术方案选择，均需先与我沟通确认，获得明确指令后再执行。
2.  **步骤化沟通与确认**：每一步开发操作（包括但不限于代码编写、功能迭代、配置调整、模块设计）均需以**明确的步骤形式**展开，且每一步执行前必须：
    - 清晰说明**即将执行的具体操作内容**；
    - 等待我确认同意后，再开展后续工作；
    - 若我未明确回复，需暂停所有操作，直至收到指令。
3.  **全链路利弊透明化**：针对每一次编辑、修改或新增的内容，必须**同时清晰阐述该操作的好处与潜在坏处**，具体要求如下：
    - 好处：从项目整体进度、代码质量、功能实现、性能优化、可维护性等维度，具体说明该操作能带来的正向价值；
    - 坏处：从技术风险、兼容性、后续迭代难度、资源消耗等维度，客观分析该操作可能存在的隐患与不足；
    - 需确保利弊分析**贴合项目当前实际场景**，不泛泛而谈，不夸大或缩小影响。
4.  **基于整体思路的最优建议**：需基于我预先明确的**项目整体核心思路、目标、技术栈规划、长期迭代方向**，给出**唯一且最优的建议**，具体要求：
    - 建议需紧扣项目整体目标，避免偏离核心方向；
    - 仅提供**最优解**，不提供多个备选方案（除非我明确要求对比）；
    - 建议需结合技术原理与项目实际，说明为何该方案是最优选择，同时呼应前文提到的利弊分析。

## 协作行为规范
1.  沟通语言需**精准、简洁、专业**，避免模糊表述、歧义性描述或主观臆断性表达；
2.  所有输出内容（包括操作说明、利弊分析、建议）需**结构化呈现**（如分点、分模块），便于快速理解；
3.  若我未明确说明项目整体思路，需先引导我梳理核心目标、技术栈、功能清单等关键信息，再开展后续协作；
4.  若我提出的需求存在技术可行性风险或与整体思路冲突，需先明确指出问题，再给出调整建议，而非直接拒绝或擅自修改。

## CI/CD 标准流程规范
> 项目开发遵循 CI/CD 标准流程，每一步执行前须经讨论确认，执行后须验证结果。

# SkyOps Agent 开发指南

本文件是 SkyOps Agent 项目根目录下的长期开发指南，供 Codex、Claude等 AI 编程代理在后续理解、设计、实现、测试和维护本项目时遵循。

SkyOps Agent 的核心不是“检测缺陷”，而是“让低空作业任务更安全、更高效、更可解释、更可复盘”。所有开发决策都应围绕低空作业运营、任务级自治、风险推演、异常重构和闭环复盘展开。

## 1. 项目定位

SkyOps Agent 是一个面向深圳低空经济与无人机产业场景的低空作业任务自治与风险推演智能体。

它不是：

- 传统无人机巡检识别系统。
- 裂缝识别工具。
- 光伏热斑检测工具。
- 电力线路缺陷识别工具。
- 自动生成巡检报告工具。
- 无人机底层飞控系统。
- 绕过监管或审批的工具。

它是：

- 低空作业任务自治 Agent。
- 低空作业风险推演 Agent。
- 无人机任务规划与异常处置 Agent。
- 企业级、园区级、城市治理场景中的低空作业安全调度 Agent。
- 任务需求与无人机、机库、飞控、载荷、巡检平台之间的智能决策层。

SkyOps Agent 要回答的问题不是“无人机会不会飞”或“图像里有没有缺陷”，而是：

- 这个低空作业任务该不该飞？
- 什么时间飞最安全？
- 应该选择什么起降点、航线、设备和任务拆分方式？
- 遇到风速突变、GPS 遮挡、图传延迟、人流聚集、禁飞区变化、电量不足等异常时怎么办？
- 如果任务无法一次完成，如何自动生成补飞计划？
- 如何让每一次低空作业都有可解释、可复盘、可持续优化的闭环？

巡检只是第一个 Demo 场景，不应把产品本体误做成“无人机巡检缺陷识别”。产品能力应可扩展到建筑外立面巡检、园区光伏巡检、工地安全巡查、应急救援、园区安防、城市治理、消防巡查、低空测绘、物流配送等低空作业场景。

## 2. 核心理念

### 2.1 从“能飞”走向“会运营”

无人机硬件、飞控、机库、图像采集、热成像、LiDAR、缺陷识别等能力已经相对成熟。SkyOps Agent 不重复造这些轮子，而是聚焦低空作业规模化落地中的安全、调度、风险、异常处置和任务闭环。

开发时应优先建设：

- 任务理解能力。
- 多源约束建模能力。
- 风险识别与推演能力。
- 可执行任务规划能力。
- 异常事件下的重规划能力。
- 任务复盘和长期优化能力。
- 评测集与量化指标。

### 2.2 任务级自治决策层

SkyOps Agent 位于三层体系的中间层：

- 上层：城市级低空监管、空域管理、UTM 平台。
- 中层：SkyOps Agent 任务自治与风险推演层。
- 下层：无人机、机库、飞控、载荷、巡检平台。

SkyOps Agent 不替代政府低空监管平台，也不替代 DJI 等厂商飞控系统。它的职责是在任务需求和执行系统之间形成可解释、可约束、可复盘的智能决策。

### 2.3 避免误做成传统巡检工具

不要把本项目做成：

- 无人机拍照识别裂缝 Agent。
- 光伏热斑检测 Agent。
- 电力线路缺陷识别 Agent。
- 自动生成巡检报告工具。

这些方向已有成熟方案，创新性不足。即使 Demo 使用“高层建筑外立面巡检”，也应把重点放在任务自治和风险推演，而不是 CV 缺陷识别本身。

正确方向是：

- 低空作业任务自治。
- 低空作业风险推演。
- 任务规划与异常处置。
- 企业或园区级低空作业安全调度。
- 面向深圳低空经济场景的任务级自治能力。

### 2.4 Agent 必须体现主动性

SkyOps Agent 不是固定流程表单，也不是普通聊天机器人。它必须体现 AI Agent 从被动响应到主动思考与自主行动的转变。

核心能力关键词必须贯穿产品、代码、测试和界面：

- 主动感知。
- 多源约束。
- 风险推演。
- 自主规划。
- 异常重构。
- 闭环复盘。
- 可解释决策。
- 低空作业运营。
- 任务级自治。
- 深圳低空经济。

Agent 应能：

- 主动补齐任务所需约束，而不是等待用户填写完整表单。
- 同时考虑天气、风速、光照、人流、空域、GPS、设备、电量、载荷、图传、安全、法规、业务优先级等因素。
- 在执行前进行 what-if 风险推演。
- 生成可执行的飞行时间窗口、起降点、航线策略、任务拆分方案、安全阈值和中止条件。
- 遇到风速突变、GPS 异常、禁飞区变化、图传延迟、电量不足、人流聚集等事件时，主动给出暂停、返航、绕行、补飞、人工接管等方案。
- 任务完成后生成复盘、风险记录、补飞计划、数据质量评分、下次优化建议，并沉淀长期记忆。

## 3. 系统边界

### 3.1 当前版本边界

当前版本可以使用模拟数据完成端到端任务推演，不要求真实无人机接入。

允许：

- 使用 mock 或 simulated data 表示天气、地图、禁飞区、人流、设备状态和异常事件。
- 使用规则、配置、LLM 或混合方式完成任务解析、风险评估、任务规划、异常处置和复盘。
- 使用简化地图或沙盘界面演示起降点、航线、风险热力图和重规划过程。
- 通过结构化 JSON 与自然语言解释共同输出任务方案。

不要求：

- 真实飞控接入。
- 真实控制无人机起飞。
- 真实 CV 缺陷识别。
- 真实地图、天气、UTM 或机库接口。

必须：

- 不要把 mock 数据伪装成真实数据。
- 界面、日志、报告和文档必须清楚标注 simulated、mock、demo 或 equivalent 标识。
- 所有安全规则应显式配置，不应只隐藏在 prompt 中。
- 所有关键决策必须保留原因、依据、权衡和替代方案字段。

### 3.2 后续扩展边界

后续可扩展对接：

- DJI Pilot 或其他无人机任务系统。
- 机库系统。
- 企业巡检平台。
- UTM 或空域管理平台。
- 真实天气 API。
- 地图或 OSM 数据。
- 巡检结果分析接口。

接入真实系统时，SkyOps Agent 仍然是任务决策辅助系统，不得承诺替代飞行许可、人工安全责任人或政府监管流程。

## 4. 安全与合规原则

SkyOps Agent 是低空作业任务决策辅助系统，而不是规避监管工具。

所有涉及飞行许可、禁飞区、人员安全、城市公共安全的场景，都必须优先遵守法规和安全约束。

系统不得：

- 建议绕过禁飞区。
- 建议忽略审批。
- 建议压低安全余量。
- 建议在人流密集区域冒险飞行。
- 将不确定数据伪装成确定事实。
- 将模型建议包装成法定许可或最终安全批准。

当数据不足或风险不确定时，必须明确输出：

> 当前信息不足，建议人工复核/暂停执行/启用保守方案。

安全策略必须保守。遇到硬约束冲突或关键数据缺失时，应优先建议人工复核、暂停任务、返航、使用保守航线或拆分补飞。

## 5. 用户体验原则

用户不应像配置工业软件一样填写大量表单，而应像对专业低空调度员说话。

系统应支持自然语言输入，例如：

- “明天上午巡检这栋楼。”
- “今天下午对园区屋顶光伏做一次热斑排查。”
- “台风前帮我检查这片工地的塔吊和围挡。”
- “今晚安排一次校园安防巡逻，避开宿舍区低空飞行。”

系统应主动补全任务细节，必要时只问关键问题：

- 是否允许拆分为两次飞行？
- 是否优先保证安全而非覆盖率？
- 是否启用保守航线？
- 是否允许人工接管？
- 是否需要生成补飞计划？

交互原则：

- 少问用户，多做推理。
- 不只给结论，还解释原因。
- 不只给方案，还给备选。
- 不只执行一次，还形成复盘。
- 所有关键决策必须可解释。

输出应区分：

- 事实输入。
- 模型推理。
- 建议动作。
- 需要人工确认的事项。

## 6. 推荐多 Agent 架构

推荐采用多 Agent 协作架构。每个 Agent 应有清晰输入、输出、职责和测试边界。不要把所有逻辑塞进单个 prompt、单个函数或单个服务里。

### 6.1 Task Understanding Agent / 任务理解 Agent

职责：

- 从自然语言中抽取作业对象、作业区域、作业目标、时间要求、精度要求、风险偏好和特殊限制。
- 判断任务类型、任务等级和需要补充的信息。
- 将模糊自然语言转为结构化 MissionTask。

输入示例：

> 明天上午巡检南山区某 180 米高办公楼外立面，重点排查幕墙裂缝和脱落风险，尽量避开人流高峰。

输出应包括：

- 作业对象：高层建筑外立面。
- 作业目标：幕墙裂缝、脱落风险。
- 时间窗口：明天上午。
- 风险偏好：避开人流高峰，安全优先。
- 任务等级初判。
- 缺失信息清单。
- 需要人工确认的问题。

### 6.2 Environment Sensing Agent / 环境感知 Agent

职责：

- 整合天气、风速、降雨、能见度、光照、人流、建筑环境、GPS 遮挡、反光风险等信息。
- 在 mock 阶段从集中管理的模拟数据读取环境状态。
- 给出可飞窗口和环境风险解释。

输出应包括：

- 可飞时间窗口。
- 环境风险等级。
- GPS 弱信号区域。
- 人流高风险区域。
- 光照或反光风险。
- 是否建议调整作业时间。
- 数据来源标识，区分 mock 与真实数据。

### 6.3 Airspace Compliance Agent / 空域合规 Agent

职责：

- 检查禁飞区、限飞区、临时管控区、审批条件和空域合规风险。
- 对任务区域、起降点、航线和高度进行合规约束检查。
- 在不确定时建议人工复核或保守方案。

输出应包括：

- 是否可飞。
- 是否需要审批。
- 需避开的区域。
- 合规风险解释。
- 保守航线建议。
- 需要人工确认的合规事项。

### 6.4 Drone Status Agent / 设备状态 Agent

职责：

- 评估无人机、电池、载荷、相机、热成像、变焦能力、图传质量和任务续航。
- 判断任务是否需要换电、拆分或降级。
- 计算安全返航电量和最小任务余量。

输出应包括：

- 可用机型。
- 预计续航。
- 安全返航电量。
- 是否需要换电。
- 是否需要热成像或变焦相机。
- 任务是否需要拆分。
- 设备风险和限制说明。

### 6.5 Mission Planning Agent / 任务规划 Agent

职责：

- 综合任务、环境、空域和设备约束，生成可执行飞行方案。
- 将安全阈值、中止条件、任务拆分和备选方案显式化。
- 输出可供前端展示和评测系统检查的结构化 MissionPlan。

输出应包括：

- 推荐执行时间。
- 推荐起降点。
- 航线策略。
- 安全距离。
- 拍摄间距。
- 任务拆分方案。
- 备用方案。
- 中止条件。
- 关键决策解释。

### 6.6 Risk Simulation Agent / 风险推演 Agent

职责：

- 进行 what-if 推演。
- 对任务执行前和执行中的风险进行风险树建模。
- 将触发条件、风险等级变化、应急动作和重规划建议结构化。

必须覆盖的 what-if 示例：

- 如果风速升高怎么办？
- 如果 GPS 信号弱怎么办？
- 如果图传延迟怎么办？
- 如果电量不足怎么办？
- 如果临时禁飞区更新怎么办？
- 如果目标区域出现人群聚集怎么办？

输出应包括：

- 风险树。
- 触发条件。
- 应急动作。
- 重规划建议。
- 风险等级变化。
- 需要人工接管的条件。

### 6.7 Incident Response Agent / 异常处置 Agent

职责：

- 在任务执行中根据异常事件自动调整计划。
- 根据显式安全规则决定暂停、返航、绕行、备用航线、降级任务、拆分补飞或人工接管。
- 保存已采集数据并把未覆盖区域交给复盘和补飞流程。

输出动作包括：

- 暂停任务。
- 返航。
- 绕行。
- 切换备用航线。
- 降级任务。
- 拆分补飞。
- 请求人工接管。
- 保存已采集数据进入分析流程。

### 6.8 Report & Review Agent / 报告复盘 Agent

职责：

- 生成任务方案报告、风险评估报告、异常处置记录、补飞计划和下次优化建议。
- 明确区分事实输入、模型推理、建议动作和需要人工确认的事项。
- 将本次任务沉淀为可复盘、可评测、可改进的闭环数据。

输出应包括：

- 任务完成度。
- 数据质量评分。
- 风险触发记录。
- 异常处置过程。
- 未覆盖区域。
- 补飞建议。
- 人工复核清单。
- 下次任务优化建议。

工程要求：

- 模拟数据必须集中管理，方便替换成真实 API。
- 安全规则必须显式配置，优先放在规则文件、配置文件或可测试的策略模块中。
- 不要把关键安全规则只写在 prompt 里。
- 所有关键决策应保留 `reason`、`evidence`、`tradeoffs`、`alternatives`、`requires_human_confirmation` 等解释字段。
- LLM 输出必须经过 schema 校验或结构化解析，不能直接作为安全决策的唯一依据。
- 任何真实系统集成必须保留 mock adapter，确保本地测试不依赖外部系统。

## 8. 核心数据模型

## 9. 输出格式

系统应同时输出结构化 JSON 和面向用户的自然语言解释。

结构化 JSON 可包含：

```json
{
  "mission_summary": {},
  "extracted_constraints": {},
  "risk_assessment": [],
  "recommended_time_window": {},
  "launch_landing_points": [],
  "route_strategy": {},
  "safety_thresholds": {},
  "contingency_plan": {},
  "replan_actions": [],
  "review_report": {},
  "human_explanation": {
    "facts": [],
    "inferences": [],
    "recommended_actions": [],
    "human_confirmation_required": []
  }
}
```

输出要求：

- `mission_summary` 描述任务对象、区域、目标和优先级。
- `extracted_constraints` 记录时间、空域、人流、天气、设备、安全和业务约束。
- `risk_assessment` 必须包含风险等级、触发条件、缓解措施和证据。
- `recommended_time_window` 必须解释为什么推荐该时段，以及为什么避开其他时段。
- `launch_landing_points` 必须说明安全距离、人流影响和备选点。
- `route_strategy` 必须说明航线逻辑、任务拆分和覆盖率。
- `safety_thresholds` 必须可被测试，例如最大风速、最低电量、最小 GPS 置信度、最大图传延迟。
- `contingency_plan` 必须覆盖暂停、返航、绕行、补飞、人工接管等动作。
- `review_report` 必须区分已完成、未覆盖、风险触发、数据质量和下次优化建议。
- `human_explanation` 必须让非技术用户理解关键决策依据。

## 10. MVP Demo 方向

优先实现比赛 MVP Demo：

> 深圳某高层建筑外立面巡检任务自治与风险推演。

MVP 的重点不是识别幕墙裂缝，而是展示系统如何围绕高层建筑外立面巡检进行任务级自治决策。

### 10.1 示例流程

1. 用户输入自然语言任务：

   > 明天上午巡检南山区一栋 180 米高办公楼外立面，重点排查幕墙裂缝和脱落风险，要求尽量减少对行人的影响。

2. 系统解析任务目标、区域、对象、时间窗口、风险偏好和缺失信息。
3. 系统读取或模拟天气、人流、禁飞区、GPS 遮挡、无人机电量、图传状态。
4. 系统生成飞行时间窗口、起降点、航线策略、安全阈值和中止条件。
5. 用户或系统注入突发事件：

   - 风速突然升高。
   - 图传延迟增加。
   - GPS 置信度下降。
   - 临时限飞区更新。
   - 电量不足。
   - 目标区域出现人群聚集。

6. Agent 自动重规划，输出暂停、返航、绕行、降级、补飞或人工接管建议。
7. 系统生成任务复盘、补飞计划、风险解释和下次优化建议。

### 10.2 MVP 必须展示的能力

- 自然语言任务解析。
- 主动补全约束。
- 多源约束推理。
- 风险推演。
- 自主规划。
- 异常重构。
- 闭环复盘。
- 可解释决策。

### 10.3 MVP 不应投入过多的方向

- 不要优先开发复杂 CV 缺陷识别模型。
- 不要优先接入真实无人机飞控。
- 不要优先做完整企业后台。
- 不要把主要界面做成大表单配置系统。
- 不要把 Demo 讲成“自动生成巡检报告”。

## 11. 测试与评测理念

项目必须可测试、可量化，不应只讲概念。

测试对象不是无人机硬件，也不是 CV 缺陷识别，而是 Agent 在复杂约束下的任务决策能力。

建议建立低空任务仿真评测集。每个测试用例包含：

- 自然语言任务。
- 环境条件。
- 空域约束。
- 人流约束。
- 天气约束。
- 设备状态。
- 异常事件。
- 期望满足的硬约束。

测试用例应覆盖：

- 正常任务。
- 高风险任务。
- 冲突约束任务。
- 异常注入任务。
- 数据不足任务。
- 需要人工复核任务。
- 任务无法一次完成且需要补飞的任务。

### 11.1 Hard Constraint Pass Rate / 硬约束通过率

检查是否违反硬约束，例如：

- 禁飞区。
- 超风速。
- 低电量。
- 低于安全返航电量。
- 人流高峰低空悬停。
- 未审批空域。
- 超过设备能力的任务时长或载荷要求。

### 11.2 Risk Recall / 风险识别召回率

检查 Agent 是否识别出场景中埋设的风险点，例如：

- 风速上升。
- GPS 遮挡。
- 人流高峰。
- 临时限飞区。
- 电量不足。
- 强反光。
- 图传延迟。
- 能见度下降。
- 建筑高反射幕墙。

### 11.3 Plan Efficiency / 方案效率

比较 Agent 方案与人工基准方案：

- 飞行时间。
- 任务拆分次数。
- 目标覆盖率。
- 补飞面积。
- 人工介入次数。
- 安全余量。

效率评估不得以牺牲硬约束和安全余量为代价。

### 11.4 Incident Response Score / 异常处置得分

测试突发事件发生时，Agent 是否能：

- 暂停任务。
- 返航。
- 绕行。
- 保留已采集数据。
- 生成补飞计划。
- 解释风险。
- 通知人工接管。
- 记录异常处置过程。

### 11.5 Explainability Score / 可解释性得分

测试每个关键决策是否能说明依据、权衡和替代方案。

必须能回答：

- 为什么不建议 9 点飞？
- 为什么要拆成两段任务？
- 为什么触发返航？
- 为什么需要补飞？
- 为什么需要人工复核？
- 为什么选择保守航线？

## 12. 开发优先级

### Phase 1: 仿真 MVP

目标：证明 SkyOps Agent 能在 mock 环境中完成任务级自治与风险推演闭环。

范围：

- 自然语言任务解析。
- 模拟天气、人流、空域和设备数据。
- 基于规则和 LLM 的风险评估。
- 自动生成任务方案。
- 异常事件注入。
- 自动重规划。
- 任务复盘报告。

验收标准：

- 能完成 1 个深圳高层建筑外立面巡检端到端流程。
- 能输出结构化 JSON 和自然语言解释。
- 能处理至少 3 类异常事件。
- 能生成补飞计划和复盘报告。
- 能通过基础硬约束测试。

### Phase 2: 可视化 Demo

目标：让任务规划、风险推演和异常重构过程可视化。

范围：

- 地图或简化沙盘界面。
- 风险热力图。
- 起降点和航线展示。
- 多 Agent 协商过程展示。
- 异常重规划过程展示。
- 评测面板展示。

验收标准：

- 用户能从界面看到任务输入、约束来源、推荐方案、风险解释和重规划过程。
- mock 数据标识清晰。
- 关键决策可追踪到原因和规则。

### Phase 3: 评测集与指标

目标：把 Agent 能力变成可量化结果。

范围：

- 构建 30-50 个低空作业仿真场景。
- 自动评估硬约束通过率。
- 自动评估风险识别召回率。
- 自动评估异常处置得分。
- 自动评估可解释性。
- 输出整体测试报告。

验收标准：

- 每个 EvaluationCase 可重复运行。
- EvaluationResult 可稳定输出。
- 至少覆盖建筑外立面、园区光伏、工地巡查、安防巡逻等多类场景。

### Phase 4: 可扩展接口

目标：为真实低空运营系统预留接口，而不破坏仿真测试能力。

范围：

- 预留真实天气 API 接口。
- 预留地图或 OSM 接口。
- 预留无人机或机库系统接口。
- 预留 UTM 或空域平台接口。
- 预留巡检结果分析接口。

验收标准：

- 所有外部接口均有 adapter 抽象。
- mock adapter 与真实 adapter 可替换。
- 真实数据来源、时间戳和置信度可追踪。
- 外部接口失败时系统能降级到人工复核或保守方案。

## 13. 开发协作规范

后续 AI 编程代理开发代码时必须遵守：

- 优先保证模块清晰、类型明确、可测试。
- 不要把所有逻辑塞进一个大函数。
- 不要把安全规则只隐藏在 prompt 中。
- 不要让 LLM 的自由文本直接驱动高风险动作。
- 不要为了 Demo 效果牺牲安全边界。
- 不要把 mock 数据伪装成真实数据。
- 不要承诺系统可以绕过法规审批或替代人工安全责任人。
- 所有模拟数据集中管理，方便替换成真实 API。
- 所有关键决策保留解释字段，便于前端展示和测试断言。
- 测试用例覆盖正常任务、风险任务、冲突约束任务和异常注入任务。
- 输出报告时区分事实输入、模型推理、建议动作和需要人工确认的事项。

修改代码前应先理解当前目录结构、已有数据模型、测试方式和运行方式。若发现已有实现与本指南冲突，应优先提出最小修正方案，避免无关重构。

## 14. 决策优先级

当需求、体验、效率和安全发生冲突时，按以下优先级决策：

1. 法规合规和公共安全。
2. 人员安全和安全返航能力。
3. 硬约束满足。
4. 可解释和可复盘。
5. 任务覆盖率。
6. 任务效率。
7. 界面表现和演示效果。

任何阶段只要出现关键数据缺失、空域不确定、设备状态不确定或风险等级过高，系统都应输出人工复核、暂停执行或启用保守方案。

## 15. 最终目标

SkyOps Agent 的最终目标是把低空作业从“能飞”推进到“会运营”。

它应体现：

- 主动感知：主动发现任务缺失约束。
- 多源约束：同时理解环境、空域、设备、人流和业务目标。
- 风险推演：在执行前和执行中模拟风险变化。
- 自主规划：生成可执行、可解释、可测试的任务方案。
- 异常重构：遇到突发事件时主动重规划。
- 闭环复盘：每次任务都沉淀为下一次优化的依据。

任何后续开发都应不断校验一个问题：

> 这项改动是否让低空作业任务更安全、更高效、更可解释、更可复盘？

如果答案是否定的，或者只是把项目推向传统巡检缺陷识别工具，就应重新调整方向。
